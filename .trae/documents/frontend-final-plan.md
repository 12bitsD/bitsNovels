# 前端最终合并计划（Audit → 修复门禁 → 产品可达 → 结构性优化）

本计划合并两份来源：

* 审计与风险优先：`2026-04-09-frontend-implementation-audit-plan.md`

* 优化方向（视觉/类型/数据层）：`frontend-optimization-plan.md`

合并原则：

1. 先恢复 DoD 门禁与关键正确性（P0），再做结构性重构（P1/P2），避免“在红灯上重构”。
2. 优化方向必须转化为“可验收的阶段目标 + 明确边界 + 回归策略（测试/覆盖率/e2e）”。
3. 字体与排版属于视觉一致性的第一等公民：单独拆成任务条目，避免在颜色 token 改造中被吞没。

***

## 0. 当前事实（以 repo 为准）

* 前端源码规模：`workspace/apps/web/src` 共 207 个源文件（154 tsx / 52 ts / 1 css）。

* 门禁状态：当前 `npm run check` 不通过（lint/typecheck/test 覆盖率阈值均存在阻断）；e2e 通过但不足以替代 DoD。

* 产品可达性：路由层主要接入 Epic 1；Epic 2/3/5/6 大量实现与测试存在，但在路由/导航层不可达，导致“无法以产品形态验收”。

***

## 1. 目标与验收口径

### 1.1 总目标

在不破坏现有功能的前提下，把当前前端从“组件实现较多但不可交付/门禁红”提升到：

* 可交付：门禁全绿、覆盖率达标

* 可验收：S1\~S5 的已完成 Sprint 范围在 UI 上可达并可演示

* 可维护：类型/数据层/视觉 token 与字体排版统一，后续迭代成本下降

### 1.2 验收口径（硬标准）

* DoD：`npm run check` 全绿，覆盖率（全局 lines）≥ 73%。

* Spec：以 `docs/SPRINT_LOG.md` 的 S1\~S5 为“必须可达并可验收”的范围；S6 及以后标注为 out-of-scope 或 backlog。

* 视觉：对齐 `design/FRONTEND.md`（色彩/间距/组件规范/无障碍）。

* NFR：对齐 `process/CONSTRAINTS.md`（自动保存 3s、防抖、重试、撤销栈等）。

## 2. 分阶段执行（建议按顺序，不并行大重构）

### Phase A（P0）：恢复门禁 + 修复关键正确性/安全性

目标：让工程回到“可修改、可回归”的健康状态。

交付与验收：

* `npm run check` 通过

* `npm run test:web` 通过且 lines ≥ 73%

范围（必须优先）：

1. 修复 lint/typecheck 阻断项（未使用变量、测试类型、Hook 规则等）。
2. 修复编辑器自动保存链路的正确性风险（避免保存旧内容）。
3. 对齐 contract 的关键字段命名（例如 `rememberMe`）。
4. 限制调试入口仅开发环境可见（如 DevNavigation）。

说明：

* Phase A 不做大规模架构迁移，不引入新状态库/新请求库，只做“恢复健康 + 修复 correctness”。

***

### Phase B（P1）：打通“产品可达性”与验收闭环（工作台 Shell）

目标：把已有 Epic 2/3/5/6 的实现接入到可达的路由/导航中，使 S1\~S5 能按产品形态验收。

交付与验收：

* 新增/调整路由与导航后，核心页面可达：项目 → 工作台（章节树/编辑器/知识库）→ 通知中心 → 导出/备份入口。

* Playwright 冒烟用例覆盖关键跳转链路（或扩展现有 e2e）。

范围：

1. 增加工作台 Shell：三栏布局（左章节面板 / 中编辑器 / 右知识库面板）+ 顶栏（含通知铃铛）。
2. 把 Epic 6 通知铃铛集成到全局顶栏，满足“任何页面可打开通知中心”的交互要求。
3. 把 Epic 5 导出/备份/模板的入口接入到设置页或工作台的明确位置（按 spec 约定）。

***

### Phase C（P1）：视觉一致性治理（token 收敛 + 字体排版升级）

目标：解决“硬编码色值/灰阶体系混用”与“字体不够优雅/不统一”的问题，让整体更耐看、更专业。

交付与验收：

* 页面/组件使用设计 token（`bg-parchment/text-ink/border-border` 等）为主；硬编码色值显著减少并建立准入规则。

* 字体方案落地：界面字体与编辑器字体统一且更优雅；关键页面排版观感明显提升。

范围：

1. 颜色 token 收敛：替换 `bg-[#F5F0E8]`、`bg-gray-*` 等散落用法到语义 token（允许个别图表/第三方组件例外，但要收敛到可控范围）。
2. Typography（重点新增）：制定并落地“更优雅的字体方案”，覆盖：

   * UI 字体（导航、按钮、表单、卡片）

   * 编辑器正文/标题字体（写作体验为先）

   * 数字/统计等宽字体（数据感）
3. 无障碍：确保聚焦样式、对比度、键盘可达符合 `design/FRONTEND.md`。

字体方案选择原则（先定约束，再选字体）：

* 中文长文阅读优先：字形清晰、灰度均匀、长时间不疲劳

* 跨平台可用：macOS/Windows/Linux 都有合理 fallback

* 加载策略：如引入 WebFont，需有子集化/预加载/超时 fallback，避免阻塞渲染

***

### Phase D（P1）：类型集中与契约对齐（不引入第二套“真相源”）

目标：消除组件内散落类型、统一与 `specs/epic-N/contract.md` 和 `@bitsnovels/api-types` 的对齐方式，降低接口漂移成本。

交付与验收：

* 关键实体（Project/Volume/Chapter/KB\* 等）的类型引用统一

* API request/response 字段命名策略一致（camelCase vs snake\_case 有明确边界与转换点）

范围：

1. 优先使用 `@bitsnovels/api-types`（OpenAPI 生成）作为 API 类型来源。
2. 对于 UI 视图模型（VM）允许存在，但必须集中定义并明确转换边界（例如 `src/features/.../types.ts` 或统一模块出口）。
3. 禁止在组件内部重复定义契约实体形状（除非是局部 UI-only 结构）。

***

### Phase E（P2）：数据层重构评估（可选：TanStack Query）

结论：TanStack Query 是候选，但必须在 Phase A/B/C/D 完成后再评估与落地。

准入条件：

* 当前请求层与错误处理已收敛，有明确的请求取消/错误映射/重试策略

* 现有 hooks 的状态边界清晰，迁移不会引入“逻辑重复两套并存”

落地策略（如果启用）：

* 先选 1\~2 条高价值链路试点（例如 KB 列表 + 详情、通知分页）

* 试点通过后再逐模块迁移，避免“一次性全量迁移”
