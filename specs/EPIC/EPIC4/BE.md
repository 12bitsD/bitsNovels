# EPIC 4 后端要以统一 AI 调用链和上下文注入规则支撑全部写作任务

## 读者确认
- 核心读者：后端开发、AI 平台接入、产品评审
- 已具备背景：已了解章节、知识库、项目设置、Parser 的领域模型
- 最关心：AI 请求如何组装上下文、如何流式返回、如何复用项目级配置

## 问题定义
一句话问题：Epic 4 需要把多个 AI 写作能力收敛到一套可配置、可流式、可中止、可复用的服务链路，否则每个任务都会重复实现模型调用和上下文拼装。

## 核心结论
1. 后端应提供统一 AI 任务入口，再按任务类型路由 Prompt、上下文和结果解析。- 证据：US-4.1~US-4.8 覆盖续写、润色、扩缩写、对话、大纲、起名、建议、项目配置，任务类型多但底层能力一致。
2. US-4.1 的上下文注入规则必须写成明确算法，因为它同时要求当前章节、前章末尾、知识库条目与 Token 优先级裁剪。- 证据：US-4.1 AC#3 明确给出注入来源和保留优先级。
3. 流式协议要优先服务续写和对话，其它任务可复用同一通道，但结果解析策略不同。- 证据：US-4.1、US-4.4 明确要求流式输出；Diff、大纲、起名、建议需要结构化落地。

## 推荐后端架构

### Epic 4 应共用一条 AI 调用链，而不是为每个故事单独接第三方接口
- 客户端先调用统一任务创建接口，例如 `POST /api/ai/tasks`，请求体包含 `type`、章节/选区信息、任务参数。
- 服务端内部通过 `AIOrchestrator` 统一完成配置解析、上下文装配、Prompt 构造、模型调用、结果归一化。
- 上游模型调用建议统一走支持流式输出的聊天补全接口，外部提供商通过适配器封装，避免业务层感知具体厂商差异。
- 任务停止使用单独接口，例如 `POST /api/ai/tasks/:taskId/stop`，由服务端中断上游流并标记任务状态为 `stopped`。
- 任务结果读取使用统一流式通道，例如 SSE 或 WebSocket；若采用 HTTP SSE，则每个任务独立订阅最简单。

## 任务路由与输出策略

| 任务 | 上游调用方式 | 结果解析策略 | 是否推荐流式 |
| --- | --- | --- | --- |
| `continue` | Chat Completions / Responses，开启 stream | 直接累计文本片段 | 是 |
| `dialogue` | Chat Completions / Responses，开启 stream | 直接累计文本片段 | 是 |
| `polish` | 同一生成接口，可开或关 stream | 累计完整文本后计算 Diff | 建议开 |
| `expand` | 同一生成接口，可开或关 stream | 累计完整文本后计算 Diff | 建议开 |
| `summarize` | 同一生成接口，可开或关 stream | 累计完整文本后计算 Diff | 建议开 |
| `outline` | 同一生成接口，要求结构化输出 | 解析为方向列表 | 否，优先完整返回 |
| `name_gen` | 同一生成接口，要求结构化输出 | 解析为名字数组 | 否，优先完整返回 |
| `advice` | 同一生成接口，要求结构化输出 | 解析为建议列表 + 段落定位 | 否，优先完整返回 |
| `parse` | 与 EPIC2 共享配置，不在本 Epic 新增独立接口 | 走 Parser 既有解析链路 | 视 Parser 而定 |

### 不同任务应复用同一提供商适配层，只在 Prompt 与结果校验上区分
- `continue`、`dialogue` 返回自由文本。
- `polish`、`expand`、`summarize` 先拿到候选文本，再与原文做 Diff，Diff 在服务端产出，前端直接消费结构化结果。
- `outline`、`name_gen`、`advice` 要求模型输出 JSON 或可稳定解析的结构化文本，再转成统一 `AIResult`。

## US-4.1 核心：上下文注入规则

### 续写的上下文装配要严格按优先级裁剪，否则最关键的当前章节会被低价值信息挤掉
- 续写任务固定输入块：系统指令、任务指令、项目 AI 配置生效值、当前章节全文、前一章末尾 2000 字、当前章节相关知识库条目、世界观设定（US-2.10）。
- 相关知识库条目由系统自动关联，推荐按章节已关联实体、最近引用实体、世界观标签三类来源排序。
- Token 预算计算顺序应先预留系统指令和输出长度，再把剩余预算用于上下文。

### Token 超限时要执行固定裁剪顺序，这一顺序必须对外可解释
1. 永远保留系统指令、任务指令、项目级生效配置与输出预算。
2. 优先保留当前章节全文；如果仍超限，只能对当前章节做分段摘要，但必须最后裁剪，而且优先保留光标附近段落的原文。
3. 其次保留知识库条目，包含角色、设定、世界观；知识库内部再按相关度从高到低裁剪。
4. 最后保留前一章末尾 2000 字；若超限，先缩短到更短尾段，再整体移除。

### 推荐实现算法要明确，便于联调和排障
1. 收集原始上下文块，并为每块打上 `source`、`priority`、`estimatedTokens`、`required`。
2. 先加入 `required` 块：系统指令、任务指令、配置块。
3. 加入当前章节全文。
4. 按相关度依次加入知识库条目与世界观块，直到接近预算上限。
5. 最后尝试加入前一章末尾 2000 字；如果超限，逐步缩短尾段长度。
6. 若当前章节全文本身已逼近上限，则保留靠近光标的段落原文，并对较远段落做摘要化压缩。
7. 将最终上下文清单写入任务日志，便于后续解释“为什么这次模型没看到某条设定”。

## 其它任务的上下文构造

### 其它任务要按最小必要上下文装配，避免无谓消耗 Token
- `polish`：原选中文本 + 当前章节全文；原文必须单独标记，便于模型只输出替代版本。
- `expand` / `summarize`：原选中文本 + 当前章节全文；额外传入目标倾向，例如扩写到 1.5~2 倍、缩写到约 0.5 倍。
- `dialogue`：当前光标前后局部正文 + 当前章节全文 + 选定角色画像 + 场景描述。
- `outline`：已有前文章节摘要 + 当前项目知识库 + 未回收伏笔列表 + 当前空章节基础信息。
- `name_gen`：命名类型、风格/文化背景、性别、附加要求 + 项目世界观与命名风格约束。
- `advice`：当前章节全文 + 段落编号映射，要求模型按节奏、人物、描写、情节四个维度输出建议。

## 流式响应处理

### 流式协议要先解决增量、停止和失败三件事，续写体验才会稳定
- 上游模型若返回增量文本片段，服务端统一转成事件流，例如：`task.started`、`task.delta`、`task.completed`、`task.failed`、`task.stopped`。
- `continue`、`dialogue` 的 `task.delta` 可直接把文本片段下发给前端。
- `polish`、`expand`、`summarize` 即使开启流式，也应先在服务端累计完整文本，待生成结束后再计算 Diff 并下发 `AIResult`。
- 收到停止请求后，服务端应立即中断上游连接，并把当前累计内容作为部分结果保存到任务上下文，状态改为 `stopped`。
- 任何流式错误都要落成可展示错误信息，不能只断流不返回状态。

## Diff 与结构化结果生成

### Diff 结果应该由服务端统一产出，这样前端不会因为算法差异出现展示不一致
- `polish`、`expand`、`summarize` 在服务端拿到最终文本后，与原选区文本做逐词或逐字符 Diff。
- 输出统一 `AIDiffChange[]`，只暴露 `insert` 与 `delete` 两种变更类型，前端按颜色渲染。
- `outline` 输出为建议列表，每项至少包含标题、要点、涉及角色。
- `name_gen` 输出为名字数组。
- `advice` 输出为建议数组，每项至少包含维度、内容、段落位置。

## 项目 AI 配置与生效规则

### 所有 Epic 4 任务都要走同一套三级配置覆盖逻辑
- 生效优先级固定为：项目级（US-4.8）> 用户级（US-6.4）> 系统默认。
- `model`、`temperature`、`maxLength`、`parseDepth` 都按字段级继承，不能整对象覆盖。
- 保存项目配置后，新任务立即读取新配置；已启动任务继续使用创建时快照，保证执行期间配置稳定。
- `parseDepth` 虽然在 Epic 4 配置页维护，但它同时服务 EPIC2 Parser，因此配置服务必须跨 Epic 共享。

## 建议的数据流

### 后端接口最好围绕“创建任务、订阅结果、停止任务、保存配置”四个动作组织
- `POST /api/ai/tasks`：创建任务并返回 `taskId`。
- `GET /api/ai/tasks/:taskId/stream`：订阅流式事件或最终结果。
- `POST /api/ai/tasks/:taskId/stop`：停止任务。
- `GET /api/projects/:projectId/ai-config`：读取项目 AI 配置与生效来源。
- `PUT /api/projects/:projectId/ai-config`：保存项目 AI 配置。
