# Epic 5 BE 拆分 - 后端聚焦生成、打包、序列化与调度

## 目标结论
- Epic 5 的后端需要统一提供 3 套产物链路：作品导出文件、知识库 JSON、完整项目备份 ZIP。
- `US-5.1 导出` 与 `US-5.5 备份` 必须明确分层：前者只面向交稿正文，后者面向迁移/存档的完整项目数据。
- 所有长耗时操作都应建模为异步任务，并产出可过期的下载链接、历史记录和通知事件。

## 边界定义

| 能力 | 目标 | 输出格式 | 数据范围 | 典型入口 |
| --- | --- | --- | --- | --- |
| US-5.1 导出作品 | 投稿、分享 | DOCX / TXT / PDF / Markdown | 仅正文，可选备注/批注，不含知识库 | 工作台“导出” |
| US-5.4 知识库导出 | 跨项目复用世界观 | JSON | 仅知识库条目与关系 | 知识库面板 |
| US-5.5 项目备份 | 长期存档、迁移、恢复 | ZIP | 全量项目数据 | 项目设置“备份与恢复” |

## US-5.1 导出作品

### 任务模型
- 创建 `ExportTask` 记录，状态至少覆盖 `pending / generating / done / failed`。
- 导出走异步任务队列，避免阻塞编辑器请求。
- 任务完成后写回文件地址、文件大小、过期时间，并发送通知中心事件。

### 导出组装逻辑
- 范围解析器负责把 `全书 / 卷 / 章节` 选择转换为有序章节列表。
- 内容组装器只拼接正文；备注和批注作为可选附加内容写入目标格式。
- 知识库数据不进入该链路，必须与 US-5.4 分离。

### 格式生成器
- `DOCX`：支持字体、字号、行距、卷标题页、章节编号。
- `PDF`：共享排版参数，但走独立渲染器输出固定版式文件。
- `Markdown`：富文本内容转换为标准 Markdown 语法输出 .md 文件。
- `TXT`：处理编码和章节分隔策略。

### 存储与历史
- 建议存储路径：`exports/{userId}/{projectId}/{taskId}/result.{ext}`。
- 下载链接有效期固定 7 天，建议使用签名 URL 或一次性下载令牌。
- 导出历史保留全量元数据，查询时按时间倒序返回最近 10 条。
- 每个项目保存一份“最近使用配置”，用于下次默认值回填。

## US-5.2 自动备份

### 调度策略
- 每 24 小时扫描一次最近有变更的项目，为其创建自动备份任务。
- 自动备份内容与 US-5.5 完全一致，不是轻量正文导出。
- 归档项目直接跳过，取消归档后恢复正常扫描。

### 保留与通知
- 每个项目只保留最近 7 次自动备份，超出后删除最旧文件与元数据。
- 自动备份完成和失败都要发送通知中心事件。
- 列表接口返回时间、文件大小、下载信息，供“备份与恢复”Tab 展示。

### 存储路径
- 建议自动备份路径：`backups/{userId}/{projectId}/auto/{backupId}.zip`。
- 手动备份与自动备份建议分桶或分目录，避免权限和保留策略混淆。

## US-5.3 导出模板

### 模板存储
- 模板为用户级资源，不绑定单个项目。
- 模板数量上限 20，写入前需要后端校验。
- 模板持久化内容只保存格式和导出选项，不保存范围。

### 模板接口
- 支持创建、列表、重命名、删除四类接口。
- 保存模板时校验名称长度、格式合法性和选项结构。
- 选择模板时返回完整配置对象，供 FE 一次性填充。

## US-5.4 知识库导出与导入

### JSON Schema 定义
- 输出必须是结构化 JSON，而不是扁平文本。
- 根对象至少包含：`version`、`projectMeta`、`exportedAt`、`scope`、`knowledgeBase`。
- `knowledgeBase` 下按类型分组：角色、地点、道具、势力、伏笔、世界观设定、关系。
- 每个条目需要携带完整字段、原始 ID、显示名称、引用关系。

### 导出规则
- 全量导出时输出当前项目所有知识库条目和关系图谱。
- 按类型或按条目导出时，需要连带导出被引用关系，避免导入后出现悬空引用。
- 文件名固定为 `{项目名}_知识库_{日期}.json`。

### 导入规则
- 导入前校验 schema 版本、必填字段和类型集合。
- 冲突检测键为“同类型 + 同名”。
- 冲突处理支持 `skip / overwrite / keep_both`；`keep_both` 需自动生成不冲突名称。
- 执行结果返回导入、跳过、覆盖、重命名数量，供 FE 展示 toast。

## US-5.5 项目备份与恢复

### 先明确与 US-5.1 的区别
- `US-5.1 导出` 只生成可阅读/投稿的正文文件，不要求可逆恢复。
- `US-5.5 备份` 生成完整项目快照，必须可用于恢复、迁移和长期存档。
- 因此备份 ZIP 需要包含结构化 JSON、原始文本、版本信息和兼容性标记。

### 备份生成
- 手动备份与自动备份共用同一打包器，但来源不同：前者由用户触发，后者由调度器触发。
- 打包器复用 US-5.1 的正文序列化模块和 US-5.4 的知识库序列化模块。
- 输出 ZIP 文件名为 `{项目名}_完整备份_{日期}.zip`。
- 建议手动备份路径：`backups/{userId}/{projectId}/manual/{backupId}.zip`。

### ZIP 包内容
- 项目基础信息、写作目标、AI 配置。
- 全部卷章结构和正文原始文本。
- 知识库全部条目及关系图谱。
- 全部章节快照、章节备注、批注。
- 备份 manifest、版本号、导出时间、数据计数。

### 恢复流程
- 上传 ZIP 后先做结构完整性校验，再做版本兼容性校验。
- 支持两种模式：`创建为新项目` 和 `覆盖当前项目`。
- 覆盖模式必须在事务边界内执行，避免半覆盖状态。
- 旧格式兼容至少 2 个主版本，建议通过 `manifest.version` 做版本分派。

### 校验与错误处理
- 缺失关键文件、JSON 结构损坏、版本超出兼容范围时，统一返回“备份文件无效或已损坏”。
- 恢复前先计算摘要：项目名、总字数、章节数、知识库条目数、备份日期。
- 恢复成功后写入操作日志，便于审计和问题追踪。

## 推荐模块划分
- `export-task-service`：创建任务、更新进度、发通知、生成下载链接。
- `content-exporter`：正文范围解析、备注/批注拼装。
- `format-generators`：DOCX、PDF、TXT、Markdown 独立生成器。
- `kb-transfer-service`：知识库 JSON 导出、schema 校验、冲突合并。
- `project-backup-service`：全量数据聚合、ZIP 打包、恢复导入。
- `backup-scheduler`：24 小时轮询、变更检测、保留清理、归档跳过。

## 覆盖检查
| 用户故事 | BE 是否覆盖 | BE 关注点 |
| --- | --- | --- |
| US-5.1 | 是 | 异步任务、正文组装、4 种格式生成、历史、下载链接 |
| US-5.2 | 是 | 24 小时调度、仅变更项目、保留 7 份、归档跳过 |
| US-5.3 | 是 | 用户级模板存储、CRUD、20 个上限、结构校验 |
| US-5.4 | 是 | JSON schema、导出命名、导入校验、冲突处理 |
| US-5.5 | 是 | 全量 ZIP 打包、恢复模式、版本兼容、技术复用 |
