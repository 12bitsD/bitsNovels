# TDD 工作流指南

本文档约定 bitsNovels 项目的测试驱动开发（TDD）工作流程。遵循红绿重构循环，每个用户故事（US）必须先有测试、后有实现，确保代码可维护、可验证。

## 读者确认

- **核心读者**：FE Agent 和 BE Agent（以及参与开发的人类开发者）
- **已具备背景**：了解 bitsNovels 是 AI 辅助长篇小说写作平台，使用敏捷开发、2 周一个 Sprint
- **最关心**：每个 US 的具体测试写法、文件放哪里、什么时候算做完

## 问题定义

一句话问题：如何让前后端 Agent 在并行开发时，通过统一的 TDD 流程保证代码质量和接口一致性？

## 核心结论

1. **红绿重构循环是铁律** —— 先写测试（红）、再写实现（绿）、最后优化（重构），跳过任何一步都会积累技术债务
2. **context.md 是前后端契约** —— 数据结构、接口定义必须先在 context.md 中约定，再各自写测试
3. **DoD 清单逐项打勾** —— 每个 US 完成前必须通过代码审查、测试通过、文档更新三项检查

## TDD 循环

每个用户故事的开发遵循以下三步循环：

```
红（Red）→ 绿（Green）→ 重构（Refactor）
```

### 第一步：红（写测试）

在写任何实现代码之前，先写测试。测试描述了你期望的功能行为。

- FE Agent：写组件测试、Hook 测试、集成测试
- BE Agent：写单元测试、API 测试、数据库测试
- 双方同时更新 `context.md` 中的数据结构定义

### 第二步：绿（写实现）

写最少的代码让测试通过。不要过度设计，先让测试变绿。

- 实现代码放在 `src/features/epicN/`（前端）或 `server/routes/epicN/`（后端）
- 确保测试全部通过后，再进行下一步

### 第三步：重构（优化）

测试通过后，优化代码结构。重构时测试必须保持绿色。

- 提取重复代码
- 优化命名
- 简化逻辑

### 前端测试

```

├── features/
│   ├── epic1/
│   │   ├── __tests__/           ← Epic1 的测试文件
│   │   │   ├── ProjectList.test.tsx
│   │   │   └── useProject.test.ts
│   │   ├── components/
│   │   └── hooks/
│   ├── epic2/
│   │   └── __tests__/
│   └── epic3/
│       └── __tests__/
```

### 后端测试

```
server/
├── routes/
│   ├── epic1/
│   │   ├── __tests__/           ← Epic1 的 API 测试
│   │   │   ├── project.test.ts
│   │   │   └── auth.test.ts
│   │   └── index.ts
│   ├── epic2/
│   │   └── __tests__/

```

### 测试命名规则

- 测试文件：`[被测模块].test.ts` 或 `[被测模块].spec.ts`
- 测试描述：用中文描述场景，例如 `describe('项目列表')` + `it('空状态时显示引导文案')`

## 谁写哪些测试

| Agent | 测试类型 | 覆盖范围 | 不覆盖 |
|-------|----------|----------|--------|
| FE Agent | 组件测试、Hook 测试、E2E 测试 | UI 渲染、用户交互、状态管理 | 不测试后端 API 实现细节 |
| BE Agent | 单元测试、集成测试、API 测试 | 业务逻辑、数据持久化、接口契约 | 不测试前端 UI 渲染 |

**关键原则**：双方共同维护 `context.md`，里面的数据结构就是前后端的契约。一方修改必须先通知另一方。

## Definition of Done（完成标准）

每个用户故事（US）必须满足以下清单才算完成：

### 代码检查项

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| 测试覆盖率 | 新增代码行覆盖率 ≥ 80% | `npm run test:coverage` |
| 单元测试 | 所有单元测试通过 | `npm run test:unit` |
| 集成测试 | 相关集成测试通过 | `npm run test:integration` |
| 类型检查 | TypeScript 无错误 | `npm run type-check` |
| 代码风格 | ESLint/Prettier 无警告 | `npm run lint` |

### 功能检查项

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| 需求对齐 | 实现与 US 描述一致 | 对照 `specs/EPIC/N/FE.md` 或 `BE.md` |
| 约束满足 | 满足所有性能约束（见下方） | 代码审查 + 性能测试 |
| 错误处理 | 异常情况有友好提示 | 手动验证错误场景 |
| 边界情况 | 空状态、超长内容等已处理 | 单元测试覆盖 |

### 文档检查项

| 检查项 | 要求 | 验证方式 |
|--------|------|----------|
| context.md 更新 | 数据结构变更已记录 | 文件 git diff |
| 代码注释 | 复杂逻辑有注释 | 代码审查 |
| CHANGELOG | 新功能已记录 | 检查 CHANGELOG.md |

## Epic 开发顺序

按以下顺序开发，避免依赖混乱：

```
EPIC1（项目管理）  →  无前置依赖，最先做
       ↓
EPIC3（写作工作台） → 项目结构是编辑器的基础
       ↓
EPIC2（知识库）    → 依赖 EPIC3 的章节结构
       ↓
EPIC4（AI 写作）   → 依赖 EPIC2 知识库 + EPIC3 编辑器
       ↓
EPIC5（导出备份）  → 依赖 EPIC3 编辑器
       ↓
EPIC6（系统设置）  → 无业务依赖，最后做
```

### 顺序理由

1. **EPIC1 先做**：登录、项目列表是所有功能的基础，没有项目其他功能无处依附
2. **EPIC3 在 EPIC2 之前**：知识库需要解析章节内容，必须先有章节结构
3. **EPIC4 在 EPIC2/3 之后**：AI 续写需要读取知识库和编辑器内容
4. **EPIC5 在 EPIC3 之后**：导出需要编辑器的内容和结构
5. **EPIC6 最后**：设置页面不阻塞其他功能

## 关键约束

以下约束在代码审查时必须检查：

| 约束 | 要求 | 测试验证 |
|------|------|----------|
| Agent 文件读取 | 每个任务最多读取 4 个文件 | 代码审查检查 |
| 自动保存 | 输入停止后 3 秒触发保存 | 单元测试模拟计时 |
| 撤销栈 | 支持 ≥ 200 步撤销 | 集成测试验证 |
| 解析超时 | Parser 超过 60 秒自动重试，再失败则报错 | API 测试模拟超时 |
| 快照限制 | 每章节自动快照最多 50 个，手动快照无限制 | 数据库约束 + 单元测试 |
| 回收站保留 | 删除内容保留 30 天 | 定时任务测试 |

### 约束详细说明

**Agent 读取限制**

- 每个开发任务，Agent 最多同时读取 4 个文件
- 超出限制需要拆分为多个任务
- 设计时考虑模块化，减少跨文件依赖

**自动保存 3 秒防抖**

```typescript
// 伪代码示例
const save = debounce(() => {
  api.saveChapter(content);
}, 3000);

// 每次输入重置计时器
editor.onChange(() => {
  save();
});
```

**撤销栈 200 步**

- 编辑器必须维护操作历史
- 达到 200 步时，最早的记录被丢弃
- 需要测试边界情况：刚好 200 步、超过 200 步

**解析超时处理**

```
用户触发解析
    ↓
开始计时（60 秒）
    ↓
超时？→ 是 → 自动重试 1 次
    ↓ 否        ↓
完成 ←────── 还超时？→ 是 → 显示错误提示
            ↓ 否
           完成
```

**快照限制**

- 自动快照：每章节最多 50 个，超出时删除最旧的
- 手动快照：无数量限制
- 数据库层面设置约束防止溢出

**回收站 30 天保留**

- 删除的项目/章节进入回收站
- 30 天后自动永久删除
- 需要定时任务清理，每天执行一次

## 并行开发协调

FE Agent 和 BE Agent 并行工作时，通过以下机制保持一致：

### 1. context.md 是契约

```
specs/EPIC/
├── EPIC1/
│   ├── FE.md         ← FE Agent 读
│   ├── BE.md         ← BE Agent 读
│   └── context.md    ← 双方共同维护（数据结构和接口）
```

- 数据结构变更必须先更新 context.md
- 接口定义变更必须先更新 context.md
- 任何一方修改 context.md，必须通知另一方

### 2. Mock 数据先行

- BE 未就绪时，FE 使用 Mock 数据开发
- Mock 数据必须符合 context.md 定义的接口
- BE 完成后，替换为真实 API

### 3. 每日同步

- 每天结束前，双方确认 context.md 是否一致
- 发现不一致立即对齐，不要累积
